# 03 · 匹配引擎

> 文档目的：把「标签打分 → 候选池 → 向量重排 → 刷新选批」整条流水线讲清楚，方便后续调权重或换向量方案。
> 涉及文件：`src/services/chatchat/{match,corpusIndex,matchSelect,matchSession,ai,types}.ts`、`scripts/build-corpus-index.py`。

---

## 1. 总览：两条匹配路径

| 入口 | 路径 | 入参 | 主函数 |
|------|------|------|--------|
| 快速匹配（问卷） | 标签筛候选 → 离线向量重排 | `filters + optionLabels` | `rankMatchesQuizTwoStage` |
| 这一刻想法（自然句） | AI 解析→标签筛+语义重排 / 无AI→纯向量 | `userText` | `rankMatchesTwoStage` / `rankMatchesByEmbedding` |

两条路径最后都走 `selectRefreshBatch` 选批 + `commitMatchBatch` 记录会话。

## 2. 标签打分（`scoreCopy`）

文件：`match.ts`

### 2.1 维度权重

```ts
const WEIGHTS: Record<string, number> = {
  mood: 3,
  scene: 2,
  style: 2,
  purpose: 1.5,
  length: 0.5,
  theme: 1,
  relation: 1,
};
```

设计思路：
- **mood 最重**（3）：情绪是「想要什么感觉」的核心，用户最敏感。
- **scene / style 次重**（2）：场景决定能不能用，风格决定喜不喜欢。
- **purpose 1.5**：用途偏软，影响排序但不该一票否决。
- **theme / relation 1**：补充筛选，不当主因。
- **length 0.5**：最弱，长度不合适也能用，只是排序靠后。

### 2.2 打分公式

对每条文案 `copy`，遍历用户 filters 的每个维度：

```
earned += hits.length * weight
max    += userTags.length * weight
```

最终 `score = max > 0 ? earned / max : 0`，归一化到 0~1。

### 2.3 精确词匹配（`containsExactTerm`）

为避免 `unhappy → happy`、`不开心 → 开心` 这种子串误命中：
- 英文词：用 `(?<![a-zA-Z])term(?![a-zA-Z])` 词边界正则。
- 中文词：用 `indexOf` 找到后检查前一个字是否在 `CN_NEGATION`（不/没/非/无/未/别/莫/勿）里，是则跳过这个位置继续找。

### 2.4 关系字段特殊处理

`relation` 在语料里是**单选字符串**（不是数组），`getCopyTags` 单独处理：

```ts
if (dim === "relation") {
  const rel = copy.tags?.relation;
  return rel ? [rel] : [];
}
```

## 3. 候选池构建（`searchCopies`）

### 3.1 渐进式放宽

```ts
const RELAX_ORDER = ["purpose", "length", "style", "scene"] as const;
```

逻辑：
1. 先用完整 filters 在全语料上打分，取 `score > 0` 的前 100 条。
2. 如果不够 100 条，按 `RELAX_ORDER` 依次把某个维度清空再试（purpose → length → style → scene）。
3. 如果还不够，全语料打分兜底（`score || 0.01`）。

**设计意图**：先尽量精确匹配，候选不够时逐步放宽**次要维度**，最后才放宽 scene（因为 scene 很重要）。

### 3.2 关键修复：问卷模式不再排除 excludeIds

`rankMatchesQuizTwoStage` 原来把 `excludeIds` 传给 `searchCopies`，刷新时已展示 id 被排除 → 候选池变小 → 触发 `RELAX_ORDER` 放宽 → 无关条目被拉上来 → **漂移**。

修复后：候选池构建不排除 excludeIds，排序结果稳定；排除已展示只在 `selectRefreshBatch` 阶段做。

> **注意**：query 模式（`rankMatchesTwoStage` / `rankMatchesByEmbedding`）目前仍排除 excludeIds，靠 `selectRefreshBatch` 的 baseline 门槛挡漂移。后续可同步改造。

## 4. 离线向量索引

### 4.1 构建（`scripts/build-corpus-index.py`）

- 输入：`tagged_corpus.json`（过滤后 ≈4733 条）
- 词表大小：2048（按 DF 取 top）
- 向量空间：char-bigram + 关键词 + 标签 + 标签中文名
- 权重：`(1 + log(tf)) * idf`，L2 归一化
- 输出：`corpus_vectors.json`（≈1.5MB）

每条文案的 term 来源：
```
bg:{bigram}      ← text 与 keyword 的 char-bigram，权重 1~2
kw:{keyword}     ← tags.keywords，权重 4
tag:{tag_id}     ← mood/scene/style/purpose/theme，权重 3
lbl:{中文}        ← tag_id 对应中文名，权重 3
```

> **设计意图**：让「标签相同」的文案在向量空间也接近，不仅靠字面重复。

### 4.2 加载（`corpusIndex.ts`）

`loadCorpusIndex()` fetch `/corpus/corpus_vectors.json`，解析成 `CorpusIndex`：
- `vocab: string[]` + `termIndex: Map<string, number>`
- `idf: number[]`
- `entries: Map<number, SparseVector>`（`SparseVector = Map<termIdx, weight>`）

单例缓存，全 App 只加载一次。

### 4.3 查询向量构建

- 问卷：`buildQueryVector(filters, optionLabels, index)`
  - 把 optionLabels（如「暗戳戳表达喜欢」「浪漫心动」）按 `lbl:` + `bg:` 入向量
  - filters 的 tag_id 按 `tag:` + `lbl:` + `bg:` 入向量
  - keywords 按 `kw:` + `bg:` 入向量
- 自然句：`buildQueryVectorFromText(userText, index)`
  - char-bigram + 按 `[，。！？、；\s]` 切短语入 `kw:`

### 4.4 相似度

`cosineSimilarity(a, b)`：稀疏向量点积，遍历较小那个 Map。

### 4.5 局限

char-bigram + TF-IDF 只能捕捉「字面重叠 + 标签共现」，**没有真正的语义理解**。
- 「我承认我好色」能匹配到 id=3669 是因为字面 + 标签重叠，不是真的懂「好色」。
- 未来可换 bge / text2vec-base-chinese 等真 embedding，把 `corpus_vectors.json` 升级成稠密向量 + 余弦。需要同时改 `build-corpus-index.py` 与 `corpusIndex.ts`。

## 5. 两阶段排序

### 5.1 问卷模式（`rankMatchesQuizTwoStage`）

```
1. searchCopies(corpus, filters, {limit:100})      → tagCandidates（不排除 excludeIds）
2. 若 < 10 条：放宽阈值；若 < 5 条：全语料兜底
3. buildQueryVector(filters, optionLabels, index)  → queryVec
4. 对每个候选：finalScore = tagScore * 0.3 + vectorScore * 0.7
5. finalizeRanked → 按 finalScore 排序，取前 RANK_POOL 条
```

**权重 0.3 / 0.7**：标签只是粗筛，向量能捕捉字面 + 标签共现的细粒度相似，所以向量占大头。

### 5.2 Query 模式 AI（`rankMatchesTwoStage`）

```
1. parseUserIntent(query) → filters（AI 返回 JSON）
2. searchCopies → tagCandidates
3. buildSemanticTerms(userText, filters) → terms（含同义词扩展）
4. computeSemanticScore(copy, userText, terms) → semanticScore
   = vocabScore * 0.7 + fullTextScore * 0.3
   - vocabScore：terms 在 text/keywords 里的命中率
   - fullTextScore：char-bigram Jaccard * 0.55 + 短语命中 * 0.45
5. finalScore = tagScore * 0.3 + semanticScore * 0.7
```

### 5.3 Query 模式 Embedding（`rankMatchesByEmbedding`）

无 AI 配置时：
```
1. buildQueryVectorFromText(userText, index) → queryVec
2. 对全语料（排除 excludeIds）算 cosineSimilarity
3. 过滤 > 0.01，排序，取前 RANK_POOL
```

这是最弱的模式，完全靠 char-bigram 向量。

## 6. AI 意图解析（`ai.ts`）

### 6.1 Prompt（摘要）

```
你是朋友圈文案检索助手。用户描述想发的朋友圈情境，你只输出 JSON 检索条件，禁止生成任何文案。
规则：
1. mood/scene/style/purpose 各选 1~2 个最贴切的 tag_id
2. keywords 提取 3~5 个能在文案库检索的中文实体词
3. semantic_keywords 再补充 3~5 个同义/近义/联想词
4. 不要输出「朋友圈」「文案」等泛词
4. avoid 填写应排除的标签
tag_id 枚举：...
输出纯 JSON：{"mood":[],...}
```

### 6.2 解析（`parseAiIntent`）

- 容错：AI 返回带 markdown 代码块时剥壳。
- `normalizeTags`：中文词经 `CN_TO_TAG` 转 tag_id，非 `xxx_yyy` 格式的丢弃。
- `avoid` 标签会从最终 filters 里反向剔除。

### 6.3 离线兜底（`parseUserIntentLocal`）

无 API 时用规则正则 + `CN_TO_TAG` 做弱解析，返回 `MatchFilters`（含 `relation: []`）。
能力远弱于 AI，仅作 fallback。

## 7. 刷新选批（`matchSelect.ts` + `matchSession.ts`）

### 7.1 会话结构

```ts
interface MatchRefreshSession {
  sessionKey: string;        // 问卷答案 key / 查询文案
  shownIds: number[];        // 已展示
  refreshCount: number;      // 已刷新次数（不含首次）
  baselineScore?: number;    // 首次匹配最高原始分，刷新绝对门槛基准
}
```

- `MAX_REFRESH_ROUNDS = 5`
- `MIN_MATCH_PERCENT = 60`
- `MATCH_BATCH_SIZE = 3`

### 7.2 选批逻辑（`selectRefreshBatch`）

```
pool = ranked.filter(!exclude.has(id))

if baselineScore 存在（刷新场景）:
  absMin = baselineScore * 0.6
  eligible = pool.filter(matchScore >= absMin)
  if eligible.length < count: return []   ← 宁可空也不漂移
else（首次场景）:
  minScore = pool[0].matchScore * 0.6
  eligible = pool.filter(matchScore >= minScore)
  if eligible.length < count: eligible = pool.slice(0, count*4)   ← 兜底

return eligible.slice(0, count)
```

### 7.3 漂移修复关键点

| 之前 | 之后 |
|------|------|
| 候选池构建排除 excludeIds → 池缩小 → 触发 RELAX_ORDER 放宽 | 候选池不排除 excludeIds，排序稳定 |
| 门槛 = 当前池最高分 × 60%（相对） | 门槛 = baselineScore × 60%（绝对） |
| 候选不足时 fallback 取 pool.slice(0, count*4) 拉无关条目 | 候选不足时返回空，UI 提示「已无更多同主题」 |

### 7.4 会话生命周期

- 首次匹配：`commitMatchBatch(session, ids, isRefresh=false, topScore)` → 记录 `baselineScore`，`refreshCount=0`。
- 刷新：`isRefresh=true` → `refreshCount++`。
- 达 5 次：UI 显示「重新开始换批」→ 清 session + 缓存 → 重新首次匹配，重建 baseline。
- sessionKey 变了（问卷答案改 / 查询文案改）：自动新建 session。

## 8. 后续可优化

- [ ] query 模式也去掉候选池的 excludeIds，与 quiz 模式一致。
- [ ] 向量索引换真 embedding（bge-small-zh 等），稠密向量 + 余弦。
- [ ] `WEIGHTS` 目前是硬编码，可做成可配置（用户偏矫情就调高 mood，偏实用就调高 scene）。
- [ ] `RELAX_ORDER` 顺序固定，可按 filters 哪个维度标签多动态调整（标签多说明用户在意，不该先放宽）。
- [ ] AI 解析可加 few-shot 示例，降低返回中文不命中 tag_id 的概率。
- [ ] baseline 门槛 60% 是经验值，可按 RANK_POOL 大小动态算（池越大可适当放宽）。
